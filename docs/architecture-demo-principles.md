# HelloTime Demo 架构原则

## 项目定位

HelloTime 是一个技术展示型 demo，不是生产模板。

这个仓库的价值，在于用多种前后端技术栈实现同一套产品语义，并让这些实现可以被并排比较、单独运行、单独测试、单独讲解。

因此，这个仓库的首要架构目标不是代码复用，而是“可控的可比性”：

- 每个实现都保持独立
- 每个实现都呈现相同的产品语义
- 仓库能够清楚地验证这些实现的“等价性”与技术差异点

## 核心原则

### 1. 独立实现是设计前提

每个后端和前端实现都必须能够在不依赖其他实现源代码的前提下单独运行。

允许共享的内容：

- OpenAPI 规范
- 设计令牌和静态资源
- 仓库级编排脚本
- 仓库级验证场景与文档

默认不应引入的共享方式：

- 后端之间共享运行时业务库
- 前端之间共享应用逻辑包
- 某个实现成为其他实现的隐式“标准答案”

### 2. 共享的是产品语义，不是运行时代码

实现可以独立，但从外部表现出来的产品行为必须一致。

以下内容属于共享产品语义，应该在所有实现中保持一致：

- API 路由与响应结构
- 错误码含义
- 时间处理与开启规则
- 管理员认证行为
- 分页语义
- 前端路由结构
- 主题持久化规则
- 管理员会话生命周期规则

技术栈之间允许的差异，应体现在框架风格、语言习惯和工具链上，而不是体现在用户可观察的产品行为上。

### 3. `spec/` 是跨实现唯一真相源

`spec/` 负责定义这个应用“是什么”，而 `backends/` 和 `frontends/` 负责定义“如何用具体技术栈实现它”。

`spec/` 至少应覆盖：

- `spec/api/openapi.yaml` 中的接口契约
- 时间校验、内容隐藏、认证语义等行为规则
- 设计令牌和视觉规则

如果两个实现出现分歧，应该通过修正实现或修正规范解决，而不是默认接受漂移。

### 4. 不共享代码，也必须共享验证

对于技术展示型 demo，架构一致性不应该依赖共享运行时代码来维持，而应该依赖共享验证来约束。

仓库应该维护一套能重复运行在任意后端或前端实现上的通用验证场景。

例如：

- 同一组后端契约检查同时跑在 Spring Boot、FastAPI、Gin、Elysia、NestJS、ASP.NET Core、Vapor、Axum、Drogon 上
- 同一组前端用户流程检查同时跑在 React、Vue、Angular、Svelte、SolidJS 上
- 同一组 API 集成测试同时验证分离式架构与全栈式（Next/Nuxt/Spring MVC）架构的行为一致性
- [规划中] 桌面端与移动端的业务逻辑核心自动化验证（使用 Appium 或 Playwright Native）

### 5. 根目录工具只负责编排，不替代本地实现的所有权

每个实现应自行拥有：

- 本地 README
- 运行命令
- 测试命令
- 构建命令
- 技术栈特定配置

仓库根级脚本只负责组织这些命令并汇总结果，不应该让某几个实现变成默认主实现。所有实现都应被视为一等公民。

### 6. Demo 应优先优化“对照价值”

每个实现都应该能清楚回答这些问题：

- 它展示的技术栈是什么
- 如何运行
- 如何测试
- 它对应了哪些共享产品语义
- 它有哪些值得观察的技术栈特征

当读者把两个实现并排看时，不应该猜测某个差异究竟是设计选择，还是意外漂移。

### 7. 跨媒介的语义“绝对一致”

对于桌面端（Tauri/SwiftUI/WinUI）和移动端（iOS）分支，虽然交互媒介、输入方式和屏幕尺寸发生了剧变，但“时间胶囊”的核心逻辑（锁定逻辑、加密状态、解锁时间计算、管理员权限边界）必须保持与 Web 原型语义的 100% 对齐。

这意味着：
- 不得因为移动端开发方便而简化 API 安全校验。
- 不得因为桌面端存储方便而改变数据的本地同步逻辑。
- 设计语言（如 Cyber-Glass 风格）应在不同平台上通过本地技术栈进行等效实现，而非简单的妥协。

## 建议的仓库职责划分

### `spec/`

负责共享产品语义。

建议包含：

- API 契约
- 错误码定义
- 时间与时区规则
- 认证与会话规则
- UI 设计令牌

### `backends/`

负责独立后端实现。

当前后端实现包括：

- `spring-boot` (Java)
- `fastapi` (Python)
- `gin` (Go)
- `elysia` (Bun/TS)
- `nest` (Node.js/TS)
- `aspnet-core` (C#/.NET)
- `ktor` (Kotlin/JVM)
- `vapor` (Swift)
- `axum` (Rust)
- `drogon` (C++)

每个后端都应该能独立启动和演示。

### `frontends/`

负责独立前端实现。

当前前端实现包括：

- `vue3-ts`
- `react-ts`
- `angular-ts`
- `svelte-ts`
- `solid-ts`

每个前端都应该能独立启动和演示。

### `fullstacks/`

负责自包含的全栈实现方案。

当前全栈实现包括：
- `next-ts` (Next.js 15)
- `nuxt-ts` (Nuxt 3)
- `spring-boot-mvc` (Spring Boot + Thymeleaf + HTMX)

全栈实现应展示“如何在一个统一的代码库中闭环产品语义”，它们通常不依赖外部 API 映射脚本。

### `desktop/` & `mobile/`

负责原生或跨平台客户端实现。

目标是展示如何将“赛博玻璃”设计系统和 OpenAPI 契约迁移到非浏览器环境：
- `desktop/tauri` (Tauri 2 + React)
- `desktop/macos-swiftui` (macOS 原生)
- `desktop/winui3` (Windows 原生)
- `mobile/ios-swiftui` (iOS 原生)

### `verification/`

负责共享验收规则和验证场景。

该目录不应包含应用运行时代码，而应包含用来证明多个实现行为等价的验证资产。

### `scripts/`

负责仓库级编排，例如：

- 运行全部后端测试
- 运行全部前端测试
- 执行验证矩阵
- 汇总仓库健康状态

## 当前最值得优先重构的方向

### 1. 深度覆盖多端验证矩阵

目前仓库虽然拥有庞大的实现矩阵，但自动化验证主要集中在 Web 端。后续应致力于将桌面端（Tauri/Native）和移动端的关键用户路径纳入根级验证体系，确保“跨平台语义一致性”不因手动测试的疏漏而失效。

### 2. 把共享行为规则从“口头约定”变成规范

像“`openAt` 必须是未来时间”或“管理员 token 为会话级”这类规则，不应该靠读取某个实现的代码来推断，而应明确写入规范层。

### 3. 区分技术差异与语义漂移

语法风格、目录结构、状态管理方式的差异是 demo 的价值所在；业务语义不一致则是架构缺陷，除非明确声明这是刻意展示的差异。

### 4. 让每个实现都具备自解释能力

每个实现都应补充以下内容：

- 技术栈概览
- 运行 / 构建 / 测试命令
- 环境变量说明
- 当前合规状态
- 已知缺口（如果存在）

## 架构评审检查表

评估后续变更时，可以用下面几个问题进行判断：

1. 这个变更是否保持了实现独立性？
2. 这个变更是否保持了共享产品语义？
3. 真相源是否写在 `spec/` 中，而不是藏在某个实现里？
4. 这个变更是否可以被跨实现验证？
5. 这个仓库是否仍然容易比较、容易演示？

如果其中任一问题的答案是否定的，那么这个变更大概率在削弱 demo 的架构质量。
