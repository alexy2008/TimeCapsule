## 3. 目录结构与组件设计思路

### 组件 vs 视图

这个项目把 UI 代码分成两类：

**视图（Views）**：对应一个完整页面，绑定路由。比如 `CreateView` 对应 `/create`
路径，它负责页面级别的数据获取和布局。

**组件（Components）**：可复用的 UI 块，不感知路由。比如 `CapsuleForm`
可以在任何视图里使用。

> 这个划分是项目的架构约定，不是 [框架名] 的要求。
> [框架名] 本身不区分"视图"和"组件"。

### API 层与类型定义

`api/index.ts` 和 `types/index.ts` 在**五个前端实现里完全相同**，
和框架无关。这样设计的好处是：切换框架时，API 调用逻辑不需要重写。

## 撰写原则

1. **以用户操作为主线** — 第 5 章追踪"创建胶囊"的完整流程，让读者跟着用户行为走，而不是跟着代码结构走。
2. **框架无关的代码放前面** — `api/index.ts` 和 `types/index.ts` 在五个前端完全一致，强调这一点帮助读者理解哪些是框架特有的、哪些是通用的。
3. **对比式说明** — 每个框架的独特写法都要和其他框架做一句话对比，帮助有经验的读者快速定位差异。
4. **代码即文档** — 每个代码块都来自项目真实代码，带文件路径，读者可以直接跳转查看上下文。
5. **不写扩展练习** — 这篇文档是导读，不是教程。读者读完应该能独立探索，不需要额外的练习题。
