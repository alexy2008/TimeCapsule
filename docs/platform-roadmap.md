# HelloTime 平台扩展路线图

本文记录 HelloTime 仓库未来的平台扩展规划，目标是把仓库从“多前端 + 多后端 + 多全栈”进一步扩展为“多语言后端、多范式全栈、桌面端和移动端并列对照”的技术展示矩阵。

## 1. 规划目标

未来的总体目标如下：

| 类别 | 当前数量 | 规划数量 | 说明 |
| :--- | ---: | ---: | :--- |
| 后端实现 | 6 | 10 | 在现有后端矩阵基础上继续扩充语言和框架覆盖 |
| 浏览器前端 | 4 | 4 | 不再新增，保持稳定对照组 |
| 全栈实现 | 3 | 5 | 继续补足经典全栈路线 |
| 桌面端 | 0 | 5 | 新增桌面端平台矩阵 |
| 移动端 | 0 | 5 | 新增移动端平台矩阵 |

最终目标不是简单增加项目数量，而是让仓库形成一套可以并排阅读、并排运行、并排验证的技术路线图。

## 2. 总体原则

### 2.1 语义统一，技术分化

所有新增实现都必须遵守同一套产品语义：

- 同一套胶囊业务
- 同一套 API 契约
- 同一套时间规则
- 同一套管理员行为
- 同一套技术栈展示规则

但在具体实现方式上允许差异：

- 语言特性不同
- 框架组织方式不同
- 目录结构不同
- 状态管理方式不同
- UI 载体不同

### 2.2 `spec/` 继续作为唯一真相源

新增平台不应绕开 `spec/` 自行定义行为，而应继续围绕：

- `spec/api/openapi.yaml`
- 时间与时区规则
- 认证与会话规则
- 设计令牌与视觉规则

来实现自己的版本。

### 2.3 每个实现都要能独立运行

新增的后端、全栈、桌面端、移动端都应当：

- 自带独立运行入口
- 自带独立 README
- 自带独立测试或验证方式
- 能被仓库级脚本统一编排

## 3. 后端扩展规划

### 3.1 目标

后端从 5 个扩展到 10 个，继续补足语言与框架覆盖面。

### 3.2 建议新增的 4 套后端

| 新增方向 | 推荐框架 | 价值 |
| :--- | :--- | :--- |
| Swift | Vapor | 展示 Swift 服务端路线 |
| Kotlin | Ktor | 展示 Kotlin 轻量后端 |
| Rust | Axum | 展示 Rust 的类型与性能路线 |
| C++ | Drogon | 展示 C++ Web 后端能力 |

### 3.3 推荐接入顺序

优先建议：

1. ASP.NET Core
2. Ktor
3. Vapor
4. Axum
5. Drogon

原因是：

- ASP.NET Core 已经作为新增后端接入完成，可作为后续新增后端的模板
- Ktor 和 Vapor 更容易快速对齐现有 REST 语义
- Axum 和 Drogon 更能补足语言路线差异

### 3.4 后端统一要求

所有后端实现都应继续提供：

- `GET /api/v1/health`
- `POST /api/v1/capsules`
- `GET /api/v1/capsules/{code}`
- `POST /api/v1/admin/login`
- `GET/DELETE /api/v1/admin/capsules`

并且继续遵守：

- 胶囊码 8 位
- 未到时间隐藏内容
- 管理员认证独立于普通用户
- 返回统一响应格式

## 4. 全栈扩展规划

### 4.1 目标

全栈从 3 套扩展到 5 套，继续补足经典服务端全栈路线。

### 4.2 当前与新增路线

| 类型 | 实现 | 重点表达 |
| :--- | :--- | :--- |
| 现代 React 全栈 | Next.js | App Router、Route Handlers、middleware、服务端壳 |
| 现代 Vue 全栈 | Nuxt | `pages/`、`useAsyncData`、Nitro、middleware |
| 经典 Java 全栈 | Spring Boot MVC | Spring MVC、Thymeleaf、HTMX、Session |
| 新增 | Laravel | PHP 全栈、Blade、Livewire、传统服务端增强 |
| 新增 | Rails | Ruby 全栈、ERB、Hotwire、约定优先 |

### 4.3 推荐接入顺序

优先建议：

1. Rails
2. Laravel

原因是：

- Rails 最能体现“约定优于配置”的全栈哲学
- Laravel 最能体现现代 PHP 全栈体验
- 它们和现有 Next / Nuxt / Spring MVC 能形成明显的路线差异

### 4.4 全栈统一要求

新增全栈实现要做到：

- 自带后端能力，不依赖 `localhost:8080`
- 仍然遵守统一 API 与业务语义
- 保持独立 README 与运行入口
- 与仓库验证脚本接入同一套主流程

## 5. 桌面端扩展规划

### 5.1 目标

桌面端从 0 扩展到 5 个，形成独立的平台矩阵。

### 5.2 建议实现

| 类型 | 建议实现 | 价值 |
| :--- | :--- | :--- |
| macOS 原生 | SwiftUI | 展示 Apple 生态原生体验 |
| Windows 原生 | WinUI 3 + C# | 展示 Windows 原生界面与系统集成 |
| 跨平台桌面 | Tauri | 展示轻量桌面壳和 Web 技术结合 |
| 跨平台桌面 | Electron | 展示成熟桌面壳和 JS 生态 |
| 跨平台桌面 | Flutter Desktop | 展示跨平台 UI 和单套渲染体系 |

### 5.3 桌面端建议特点

桌面端应该尽量表现出桌面软件的习惯，而不是浏览器壳：

- 窗口尺寸与菜单栏
- 本地文件和配置存储
- 原生日期与时间组件
- 快捷键
- 系统主题同步
- 多窗口或托盘能力（视平台而定）

### 5.4 推荐接入顺序

优先建议：

1. Tauri
2. Electron
3. Flutter Desktop
4. SwiftUI macOS
5. WinUI 3

这样可以先补充跨平台桌面能力，再进入原生平台。

## 6. 移动端扩展规划

### 6.1 目标

移动端从 0 扩展到 5 个，形成独立的移动平台矩阵。

### 6.2 建议实现

| 类型 | 建议实现 | 价值 |
| :--- | :--- | :--- |
| Android 原生 | Kotlin + Jetpack Compose | 展示 Android 原生体验 |
| iOS 原生 | SwiftUI | 展示 Apple 移动端原生体验 |
| 跨平台移动 | React Native | 展示 JS/TS 跨平台移动路线 |
| 跨平台移动 | Flutter | 展示单套 UI 渲染的跨平台路线 |
| 跨平台移动 | .NET MAUI | 展示 C# 跨平台移动路线 |

### 6.3 移动端建议特点

移动端需要更强调设备能力与交互习惯：

- 底部导航
- 原生返回行为
- 本地存储
- 原生时间选择器
- 分享与复制胶囊码
- 安全的登录态保存
- 深色模式与系统联动

### 6.4 推荐接入顺序

优先建议：

1. Flutter
2. React Native
3. .NET MAUI
4. Android 原生
5. iOS 原生

这样能先把跨平台移动的统一业务链路跑通，再补原生平台细节。

## 7. 目录建议

如果仓库继续扩展，建议新增顶层目录来区分平台类型：

```text
backends/
frontends/
fullstacks/
desktop/
mobile/
```

建议职责如下：

- `backends/`：纯后端实现
- `frontends/`：浏览器前端实现
- `fullstacks/`：前后端一体实现
- `desktop/`：桌面端实现
- `mobile/`：移动端实现

这样做能让仓库结构更加清晰，也便于后续验证脚本和文档入口维护。

## 8. 验证与文档

新增平台时建议同步补齐：

- 独立 README
- 运行命令
- 构建命令
- 测试命令
- 根级编排脚本接入
- 验证矩阵接入
- 对比文档接入

如果后续平台数量继续增加，文档结构建议分成三层：

1. 总路线图
2. 各平台类别对比
3. 各单独实现 README

## 9. 实施节奏建议

如果按风险和收益排序，建议分三阶段推进：

### 第一阶段

- 后端：`.NET`、`Ktor`、`Vapor`
- 全栈：`Rails`、`Laravel`

### 第二阶段

- 后端：`Axum`、`Drogon`
- 桌面：`Tauri`、`Electron`、`Flutter Desktop`

### 第三阶段

- 后端剩余项补齐
- 桌面原生补齐
- 移动端五套实现补齐

这样可以先建立模板，再扩展到更分散的平台。

## 10. 结论

这份规划的目标，不是把仓库变成“框架数量最多的集合”，而是把 HelloTime 变成一个可以横向比较不同平台路线的技术展示矩阵。

最终希望读者能清楚看到：

- 同一个业务在不同后端框架里怎么落地
- 同一个业务在不同全栈框架里怎么组织
- 同一个业务在桌面端和移动端里怎么适配

这才是继续扩展这个仓库最有价值的方向。
