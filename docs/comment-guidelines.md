# HelloTime 注释规范

本文定义 HelloTime 仓库的代码注释规范。由于本项目用于技术栈展示，面向初学者和技术爱好者，注释可以比常规业务项目更丰富，但仍应以“帮助理解设计与取舍”为首要目标，而不是把代码逐行翻译成中文。

## 1. 目标

- 帮助读者理解每一层代码的职责。
- 帮助读者对照不同技术栈下的同一功能实现。
- 帮助维护者快速识别业务规则、框架约束和边界处理。

## 2. 总体原则

- 默认使用中文注释，与仓库文档语言保持一致。
- 优先解释“为什么这样写”“这一层负责什么”“和其他实现如何对应”。
- 允许适度解释“做什么”，但只在初学者不容易一眼看懂时使用。
- 注释必须服务于教学，不写机械复述。
- 注释与代码发生冲突时，以代码正确性为先，并尽快修正注释。

## 3. 哪些地方应该写注释

### 3.1 建议保留的注释层级

- 文件头注释
  - 说明该文件在当前技术栈中的职责。
  - 如有必要，指出它与其他技术栈中对应的目录或抽象。
- 类、服务、组件、hook、composable、middleware 注释
  - 说明职责边界和对外能力。
- 公开函数、方法、接口注释
  - 说明输入、输出、异常、边界行为。
- 复杂逻辑块注释
  - 用于认证、时区处理、SSR/客户端差异、统一错误映射、分页回退、状态同步。

### 3.2 优先补充的教学点

- 胶囊内容在 `openAt` 之前为什么要隐藏。
- 为什么统一使用 `{ success, data, message, errorCode }` 响应结构。
- 为什么管理员态用 `sessionStorage`、cookie 或 session。
- 为什么部分逻辑放在 service / hook / composable / server utils 中，而不是页面组件里。
- 不同框架中的等价抽象。
  - Vue `composable`
  - React `hook`
  - Angular `service`
  - Svelte 状态模块
  - Next/Nuxt server utils

## 4. 哪些注释不值得写

- 能从代码字面直接读出的步骤。
  - 例如“保存到数据库”“设置 loading 为 true”“返回结果”。
- 与类型或命名完全重复的字段说明。
- 大段解释框架基础知识，但与当前代码没有直接关系。
- 为了追求注释覆盖率而写的空洞描述。

## 5. 推荐密度

- 核心教学文件：中高密度
  - `api/`
  - `service/`
  - `hooks/`
  - `composables/`
  - `controller/`
  - `middleware/`
  - `server/utils/`
- 一般页面与组件：中等密度
  - 只解释关键交互流和重要状态。
- 样式与简单模板：中低密度
  - 仅对不直观的视觉语义或结构约束做说明。

## 6. 风格约定

### 6.1 写法

- 先给结论，再补原因。
- 句子尽量短，术语尽量统一。
- 行内注释只用于解释一个不直观的值、参数或分支。
- 块注释用于解释一组操作背后的意图。

### 6.2 各语言建议

- Java
  - 公共类、公共方法优先使用 Javadoc。
  - 局部逻辑用少量行注释说明业务规则。
- TypeScript / JavaScript
  - 导出函数、hook、composable、service 建议使用块注释或 TSDoc。
- Python
  - 模块与核心函数使用 docstring。
  - 复杂分支可配合短行注释。
- Go
  - 包、导出函数遵循 Go 风格注释。
  - 内部函数仅在逻辑不直观时补充说明。

## 7. 好注释与坏注释

### 好注释

```ts
// 管理员状态使用 sessionStorage 而不是 localStorage，
// 是为了让演示项目的登录态随浏览器会话结束自动失效。
const token = sessionStorage.getItem('admin_token')
```

```java
/**
 * 详情接口和管理接口共用同一实体，但响应策略不同：
 * 普通访客在开启时间之前看不到 content，管理员始终可见。
 */
private CapsuleResponse toDetailResponse(Capsule capsule) { ... }
```

### 坏注释

```ts
// 设置 token
token = value
```

```java
// 保存到数据库
capsuleRepository.save(capsule)
```

## 8. 评审检查项

提交代码时，至少检查以下问题：

- 这条注释删掉后，读者是否仍能轻松理解代码。
- 这段复杂逻辑是否缺少“为什么这样做”的说明。
- 多技术栈实现的同类文件，注释风格是否基本一致。
- 修改业务规则后，相关注释和文档是否同步更新。

## 9. 维护策略

- 不要求一次性为所有文件补齐注释。
- 优先维护核心路径：
  - 胶囊创建
  - 胶囊开启
  - 管理员认证
  - 管理端列表与删除
  - 主题切换
  - 技术栈展示
- 对多套实现中的共性规则，优先放在 `docs/` 或 `spec/`，代码里只保留与当前实现强相关的说明。
